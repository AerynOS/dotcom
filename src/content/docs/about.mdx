AerynOS is the collective output of brand new tooling built from the ground up to deliver a Linux distribution that focuses on reliability, proven performance and efficiency. Our ground up approach has allowed us to deliver on atomic updates without requiring reboots or having to utilize containers for package delivery. Part of our design philosophy is to deliver flexibility and customizability to end users. Examples of this includes:

#### Imperative vs Declarative

By default, AerynOS utilizes an imperative approach for package management. This means that users have to explicitly specify the actions they want to take, such as installing or removing packages. 

AerynOS also provides a declarative approach using our system-model feature. With this approach, users can define the desired state of their system and let the system automatically figure out the necessary actions to achieve that state.

#### Rolling release vs Fixed release model

As standard, AerynOS utilizes a rolling release model. Package updates are tested locally and then recipes are sent to our build system build and release to our Volatile repository. This repository, as the name implies is volatile by nature so should only be used for packagers to build against the latest versions of all packages and any users who wish to help test out package updates.

By default, users will be on our Unstable repository, so called to remind users that our distribution is still in development and is not to be thought of as "stable". On a periodic basis, currently around every one or two weeks, packages will be promoted from the Volatile repository to the Unstable repository. This allows users to benefit from the latest features and improvements while still maintaining a level of testing and stability..

As package updates are promoted to our Unstable repository, each sync will come with an associated tag (by release date). Using our system-model feature, users are able to fix their system to a specific tag. This is more akin to a fixed release distribution for users who prefer a greater level of stability. Users can opt to stick with a given time for as long as they wish, or for example, to stay a couple of tags behind the current release to blend stability with the latest features.

. With this approach, users can opt to stick with the current (or any historic) tagged release more akin to Fixed Release distributions for stability.

#### Filesystem agnostic approach

AerynOS' architecture is designed to be filesystem agnostic, allowing users to benefit from the features of AerynOS without being tied to a specific filesystem. By way of example, we are able to deliver on automatic rollback capabilities without any additional filesystem-specific code such as BTRFS snapshots.

Here are the currently supported filesystems for the AerynOS root partition:

- xfs: Our default recommendation as it currently performs the best based on our distributions' specific tooling.
- ext4: Possible to use but has limitations in the number of moss states can be created and stored.
- F2FS: Currently the worst performing filesystem on AerynOS but known to work.

We may consider additional filesystems for the AerynOS root partition in the future. It should be noted that you can already mount drives with other partitions such as btrfs and bcachefs, it's just that they are not supported for the root partition itself.

## Using AerynOS

The current recomendation is to use AerynOS in a virtual machine or on a spare dedicated machine. We don't recommend using AerynOS on your primary machine due to our alpha state and risk of something going wrong.

Our tooling is designed to prevent or recover from issues that may arise during normal usage. However, we cannot guarantee that everything will work perfectly, and we encourage users to report any issues they encounter so that we can improve the system. Head over to our [Community](/community) page for the best ways to get involved and contribute to the project.

## For Contributors

Want to be part of something amazing? AerynOS is a community-driven project where everyone can make a meaningful impact. Our powerful yet simple boulder recipes make packaging accessible to newcomers, while our clean stateless design and predictable `/usr` layout helps you understand exactly how changes will impact the system. 

Whether you're interested in packaging, development, documentation, testing, or community support - check out our [docs](https://aerynos.dev) to see how easy it is to join our growing community and help shape the future of Linux distributions. 🛠️
